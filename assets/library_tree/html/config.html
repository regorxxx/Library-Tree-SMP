Multi-Panel Mode (facets)
Each panel can contain either a tree, album art or facet view and don't need to be contiguous. Example to set-up Facets with 3-panel layout: Genre|Artist|Album.
Set up
  • Insert 3 Spider Monkey/JSplitter panels into the layout and add library tree to each
  • For each panel, use quicksetup and select 'Facet'
  • Set the view of the first panel to Genre, the second to Artist and the third to Album*
  • For each panel press Shift + Windows key, choose 'Configure panel...'.At 'Script' tab set the panel name of the first panel to 'Genre', second to 'Artist' and third to 'Album'
  • For the second panel set Settings > Source to Panel, then choose select panel source and enter 'Genre' (it's case-sensitive)
  • For the third panel set the source to panel, then choose select panel source and enter 'Artist' (it's case-sensitive)"
  • Check for more info about chained source notifications. If using legacy behaviour, then third panel should use 'Genre|Artist' instead
Options
  • Optionally change the second panel at Context menu > Show album art if preferred and set the type to artist & may be circular style images
  • Optionally change the third panel at Context menu > Show album art if preferred and set the type to front
  • Adjust Settings > Album art > Group level if required
  • Optionally hide the top bar search, filter button & settings button for the second and third panels
  • It's recommended to leave enabled: remember state on exit and before search; filters auto-manage scroll

Extended syntax
Note new functions only available at Library-Tree-SMP use curly braces {} instead of parentheses (). e.g. $nowplaying{}.

$colour{normal,hover,selected}
Set different colours for parts of a line of text. All colours are optional: any undefined are set to default. Colours apply after the $colour code until either another $colour code is encountered or a splitter is reached (| or ¦ in the view pattern) or a reset code is reached: $colour{}

Normal, hover & selected can either be r-g-b or a highlight or dim indicator, e.g.
$colour{r-g-b,r-g-b,r-g-b} or $colour{-,++,+++}
Hover effect requires "Hover effect (foreground): Highlight text" to be enabled (at ).
Highlight & dim indicators:
- dim text
-- dimmer text
--- dimmest text
+ bright text
++ brighter text
+++ brightest text

Format, r-g-b & -/+ style, can be mixed in same $colour{} function.

Syntax examples:
$colour{} reset all colours to defaults.
$colour{255-0-0} set normal text colour to red; set hover & selected text colours to defaults since undefined.
$colour{,,+++} set selected text colour to brightest; set normal & hover text colours to defaults since undefined.
$colour{0-255-0,++} set normal text colour to green & hover text colour to brighter; set selected text colour to default since undefined.
$colour{255-0-0,0-255-0,0-255-255} set normal, hover and selected text colours to red, green & cyan, respectively.

Example pattern:
Item counts are a special case. They don't form part of the view pattern.
Align right: drawn separately; not controlled by $colour codes; already dimmed.
End of text: effected by $colour codes. Use $colour{} at the end of the line to reset or set to a different colour if desired ($colour{--} is used in the example above).
String limitation
$nodisplay & $colour are generally unsafe inside string manipulation functions that remove a part of a string or insert a substring by position. Instead, they should be put before the string function, e.g.
Use $colour{255-0-0}$substr(str,from,to). Don't use $substr($colour{255-0-0}str,from,to).
Use $colour{0-255-0}$swapbranchprefix{%<ARTIST>%}. Don't use $swapbranchprefix{$colour{0-255-0}%<ARTIST>%}.

Sources, Views and Filters can be associated by rules so that any setting change also sets other settings by default.
Press Shift while clicking at dropdown box entries to add/delete multiple values.
• Use buttons at bottom to create/move/delete rules.

• Rules do not disallow manually changing any related setting at a later point using the menus.

• Rules are evaluated in order, from top to bottom, until any of them is matched.

• Chaining and loops are not allowed, only 1 rule can be fully evaluated (when conditions are met).

• Beware of setting mutually exclusive if/if not conditions, since rule wil never be applied.

• Multiple values are allowed for the same if/if not condition by pressing Shift while clicking at dropdown box entries.
 ∙ e.g. A rule which applies by matching any of multiple possible views.
 ∙ If conditions will match any value on list, while if not conditions will match any value not on list.
 ∙ The dropdown button will show a counter if there is more than 1 value set. For ex. 'Value... (2)'.
 ∙ Adding a value without pressing Shift will simply replace all values with selected one.
 ∙ Note already selected values also have a different text color.

• A rule which would apply a setting triggered by a change on same setting will always be skipped.
 ∙ Rules should usually only change settings different to the ones evaluated.
 ∙ e.g. A View change which matches all conditions of a rule which also changes the View will be skipped.
 ∙ e.g. If you want to set a default View for some Source, don't try to trigger the rule with View changes via menus.

• If no conditions are set ('- Any -'), rule will apply to any Source, View and Filter.

• If condition is skipped ('- Skip -'), rule will not apply when changing associated setting (even if the conditions are matched), but only when non-skipped settings change.

• Rule triggers can be switched below. Note all conditions are checked in any case, triggers only control at which events the checks are done (associated setting change).

• Place more specific rules at top, and generic ones at bottom, so the former override later ones. e.g.
 ∙ A rule at top for playlist sources which sets filter to 'Lossless'.
 ∙ A rule at bottom with no conditions to set the filter to '- None - by default.
 ∙ Any source will automatically use no filter, except playlists which will use 'Lossless'.

• Selective settings trigger for specific rules can be emulated by setting all other conditions to '- Skip -'. e.g.
 ∙ A rule with Source and View conditions set to '- Skip -' and filter to 'Lossless'.
 ∙ Changing the Source or View using the menus will never trigger the rule.
 ∙ Changing the filter using menus to 'Lossless' will trigger the rule (and apply 'Then' section).
 ∙ Note the difference with setting conditions to '- Any -', which also triggers while changing Source/View via menus (Filter being 'Lossless' still required though).

Cache
Enable the disk cache for best performance. It contains resized images that facilitate faster loading and is built as images are initially loaded. Optimum performance will be achieved when the cache is fully built. If new images are obtained or images replaced, the cache copy can be refreshed from the refresh context menu item.

Optionally cached images can be preloaded in the background. Preloading completes shortly after initialisation. After that images should always show immediately on scrolling, subject to memory available.

Multi-panel mode (facets)
Chained source notifications:

• Facets mode is enabled when setting source to panel. More info at
• Every panel will send updates to other panels when being refreshed
• No need to set multiple sources, only the most immediate parent's name
• The entire chain will be updated if needed
• Disable the setting to roll back to legacy behaviour
• For example: Genre|Artist|Album 3-panel layout
 ◦ 'Album' panel uses 'Artist' as source, and 'Artist' panel uses 'Genre'
 ◦ Any change on 'Genre' would immediately refresh the other 2 child panels
 ◦ If disabled (legacy behaviour), then third panel should use 'Genre|Artist' instead


Multiple sources:

• Multiple sources are only needed if they are fully independent
• Along chained source notifications, sources are merged and deduplicated
• Can merge selections from total different views/filters (for ex. artist and genres)
• Destination panel's sorting and display don't follow sources' settings
• If disabled (legacy behaviour), the last one selected on UI is used instead
• For example: Artist1|Artist2|Album 3-panel layout
 ◦ 'Album' uses 'Artist1|Artist2' as sources
 ◦ Selection from both 'Artist' panels are merged and used on 'Album' panel
 ◦ Legacy behaviour uses either 'Artist1' or 'Artist2', last one selected on UI, not both


Internal cache:

• To use a panel as source, its selection must be stored somewhere
• This can be done using playlists (legacy behaviour) or internal caching
• A dummy playlist is required per panel with legacy behaviour
• Internal caching requires methods available only at latest JSplitter and SMP versions
• if not supported, it fallbacks to legacy behaviour instead
• Internal caching is preferable to avoid filling your playlist list with dummy playlists
